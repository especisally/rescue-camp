## 二、版本号管理规则

**格式**：`主版本号.次版本号.补丁版本号`，范围 0.0.1 ~ 9.9.9，每位不超过 9。

**递增规则**：
| 变更类型 | 版本号变化 |
|---------|-----------|
| 修复 Bug | 补丁版本号 +1（1.2.3 → 1.2.4）|
| 新增功能（兼容） | 次版本号 +1（1.2.3 → 1.3.0）|
| 不兼容变更 | 主版本号 +1（1.2.3 → 2.0.0）|

**同步更新要求**：版本号变更时，必须同步更新以下位置：
- [ ] `project.config.json` — `version` 字段
- [ ] 界面显示的版本号（关于页面/设置页）
- [ ] 日志文件（`docs/log/everything.md` + 对应功能日志）


## 三、改动日志管理规范

### 日志目录结构
```
docs/
├── log/
│   ├── everything.md          # 所有改动的简述汇总（每次新增一条）
│   ├── feature-xxx.md         # 各功能模块详细日志
│   └── bugfix-xxx.md          # Bug 修复详细日志
└── README.md
```

### 日志记录规则
1. **新增功能模块** → 在 `docs/log/` 中创建 `功能模块名.md`
2. **已有模块** → 在对应文件中**新增**记录，保留所有历史
3. **每次改动后** → 在 `everything.md` 中**新增一条简述**

### 日志内容格式
```markdown
# [功能模块名称] 改动记录

## [版本号] - [YYYY-MM-DD]

### 改动类型: [新增/修改/修复/优化]

#### 改动内容
- 具体修改了什么

#### 影响范围
- 影响的功能列表

#### 改动原因
- 为什么要做这个改动

#### 相关文件
- `pages/xxx/xxx.js` - 修改了什么

#### 测试验证
- 测试结果

#### 注意事项
- 兼容性问题 / 配置变更
```

### everything.md 条目格式
```markdown
## [版本号] - [日期] - [功能模块]
- 简述改动内容
- 详细日志见：`[功能模块].md`
```


## 四、Bug 修复流程（AI 执行顺序）

### 步骤 1：问题确认
- [ ] 复现 Bug（记录复现步骤）
- [ ] 分析控制台报错信息
- [ ] 确定根本原因
- [ ] 记录当前状态

### 步骤 2：定位问题代码
- [ ] 在 VSCode 中搜索相关代码
- [ ] 查看调用链路
- [ ] 理解上下文逻辑

### 步骤 3：修复代码
- [ ] **最小化修改**（只改必要的代码）
- [ ] 添加必要的日志/空值检查
- [ ] 确保不破坏其他功能

### 步骤 4：验证修复
- [ ] 微信开发者工具中测试
- [ ] **真机调试**验证（必须）
- [ ] 回归测试相关功能
- [ ] 检查是否引入新问题

### 步骤 5：提交代码
**Commit Message 格式**：
```
fix: [简要描述]

Bug原因: [详细说明]
修复方法: [详细说明]
影响范围: [说明影响的功能]
测试验证: [说明如何验证修复]
```


## 五、AI 行为准则

### 🚫 绝对禁止

1. **禁止不先分析就直接修改代码** — 必须先理解代码逻辑和上下文
2. **禁止删除现有功能代码** — 任何功能代码都需保留，即使看起来未使用
3. **禁止重构与当前任务无关的代码** — 只修改最小必要范围
4. **禁止提交未经真机测试的修复** — 必须真机验证通过
5. **禁止修改其他开发者的代码风格** — 保持代码风格一致
6. **禁止删除调试日志** — 可调整日志级别，不可删除

### ✅ 允许的行为

1. **添加调试日志** — 在关键路径使用 `console.log`/`console.error`
2. **最小化修复 Bug** — 只修改导致 Bug 的代码行
3. **添加空值检查** — 防御性编程，避免运行时错误
4. **添加异常处理** — `try-catch` 捕获并记录异常
5. **添加注释和文档** — 更新过时注释，添加新说明
6. **优化代码（保持功能不变）** — 必须在测试通过后进行

### 🔄 修改前必须做的检查

1. **阅读完整文件** — 理解设计意图，识别潜在副作用
2. **理解依赖关系** — API 依赖、组件依赖、全局状态
3. **查看调用链** — 追踪方法被谁调用，调用了谁
4. **评估影响范围** — 分析修改波及范围
5. **搜索相似问题** — 参考已有解决方案


## 六、开发工作流程

### 日常开发流程
```
VSCode 编写代码
         ↓
保存文件（自动触发微信开发者工具热更新）
         ↓
微信开发者工具预览/调试
         ↓
真机调试验证
         ↓
提交代码
```

### 调试技巧
1. **在 VSCode 中编码**，使用微信开发者工具预览
2. **真机调试**：微信开发者工具 → 真机调试 → 扫码预览
3. **查看日志**：开发者工具「调试器」Console 面板
4. **网络请求**：开发者工具「调试器」Network 面板
5. **存储查看**：开发者工具「调试器」Storage 面板

### 服务器操作
- 服务器 `www.yjjyzxy.top` 使用宝塔面板管理
- 如需在服务器操作（部署 API、修改配置等），请告知 AI
- AI 会协助提供操作指令或配置文件

### 小程序发布流程
1. 在微信开发者工具中点击「上传」
2. 填写版本号和版本描述
3. 登录微信公众平台 → 版本管理 → 提交审核
4. 审核通过后发布

## 九、常见问题处理

| 问题 | 解决方案 |
|------|---------|
| **版本号如何升？** | Bug修复→补丁+1；新功能→次版本+1；不兼容→主版本+1 |
| **紧急修复？** | hotfix分支 → 最小修改 → 真机测试 → 更新版本号 → 提交 |
| **多模块改动？** | everything.md 整体记录 + 各模块日志分别记录 |
| **API 请求失败？** | 检查域名配置、HTTPS证书、服务器状态 |
| **真机无法调试？** | 检查手机与电脑在同一网络，开启调试模式 |
| **需要服务器操作？** | 告知 AI，提供宝塔面板操作指令 |

## 十一、附录

### Commit Message 模板
```
<type>(<scope>): <subject>

Bug原因:
- <reason>

修复方法:
- <solution>

影响范围:
- <scope>

测试验证:
- <test result>
```

### Type 类型说明
| Type | 说明 |
|------|------|
| `feat` | 新功能 |
| `fix` | Bug 修复 |
| `docs` | 文档更新 |
| `style` | 代码格式（不影响运行）|
| `refactor` | 重构（不是新功能也不是修 Bug）|
| `perf` | 性能优化 |
| `test` | 测试相关 |
| `chore` | 构建/工具配置 |


**注意**：本文档基于项目开发规范改编，适用于 VSCode + Claude Code + 微信开发者工具的微信小程序开发环境。所有 AI 助手必须遵守以上规范。
